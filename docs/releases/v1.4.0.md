# Navne v1.4.0

**Released:** 2026-04-13 · [Technical changelog](../../CHANGELOG.md#140---2026-04-13)

---

## What's new

Version 1.4 adds bulk indexing. There's now a page under Tools → Navne Indexing where you can run the entity pipeline across your archive — catching up on old posts, re-running after you've updated the definition list, or cleaning up after a rough stretch of API errors. None of this happened on save; it happens when you ask for it.

---

## Three run types

**Index new** processes only posts that have never been through the pipeline. Use this when you've just installed the plugin and want to work through your backlog, or when you've added a new post type to the configuration.

**Re-index all** runs every matching post regardless of prior status. Use this after a significant change to your org definition list. The definition list primes the LLM's disambiguation — if you've added local context that wasn't there before, re-indexing applies it to everything.

**Retry failed** re-runs only the posts that errored in a previous run. The run history at the bottom of the page shows the last ten runs; when a completed run has failures, a Retry button appears on its detail page. This is the right tool after a rate-limit spike.

---

## Per-run mode picker

Every run has its own mode setting — Safe, Suggest, or YOLO — independent of the global Operating Mode on the Settings page. This matters because the use case for bulk indexing is almost always different from how you run on new posts day to day.

If your daily operating mode is Suggest, you might still want to run Re-index all in YOLO mode after a definition-list overhaul, letting the LLM auto-approve high-confidence entities across the archive without building up a suggestion backlog. The run-level picker makes that possible without touching your global setting.

---

## Safe mode bulk behavior

This is the part worth paying attention to if you run a shop with a curated entity list.

When a bulk run uses Safe mode, the pipeline runs the full NER and LLM disambiguation process as normal — but at the end, it checks every detected entity against your `navne_entity` taxonomy. Entities already in the taxonomy get tagged. Entities that aren't get dropped silently, with no pending suggestions created.

In practice this means you can retroactively apply your approved entity list to your full archive without any risk of creating unreviewed terms. New entities don't surface as suggestions; they just don't link. The whitelist is the gate, and nothing gets through it without a human adding it to the taxonomy first.

If your whitelist is empty when you start a Safe mode run, the indexing page tells you — the preview shows a notice before you commit to the run.

---

## Live progress and cancellation

The run detail page polls the REST API every two seconds and updates the status, processed count, and failed count in place. If you start a run and notice a configuration mistake, there's a Cancel button. Cancelling stops the dispatcher from queuing new batches; jobs already dispatched to Action Scheduler will finish, but no new posts get picked up.

---

## Pacing

The dispatcher doesn't send everything to Action Scheduler at once. It claims a small wave of posts (five by default), dispatches them as async jobs, then schedules itself to fire again sixty seconds later to claim the next wave. This keeps the Anthropic API request rate well within typical rate limits and gives the queue time to drain between waves. The batch size and interval are configurable via PHP constants if your site's limits are different.

---

## What's still not in this release

- Scheduled runs (run re-indexing on a cron)
- Inline per-item retry from the run detail page
- Configurable YOLO confidence threshold
- Real-time token cost estimation (the preview page shows a rough cost estimate based on a fixed average, not actual token counts)

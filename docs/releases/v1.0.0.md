# Navne v1.0.0

**Released:** 2026-04-12 · [Technical changelog](../../CHANGELOG.md#100---2026-04-12)

---

## What this is

Navne is a WordPress plugin that detects named entities in published articles and links them to taxonomy archive pages at render time. The goal is to give newsroom editors a fast, reviewable path from "article mentions a person or organization" to "that name is linked across the site."

Version 1.0 is a proof of concept. It validates the full pipeline end-to-end in Suggest mode — the editor reviews every suggestion before anything gets linked. Nothing is automatic.

---

## How it works

When an article is published or updated, the plugin dispatches an async job via Action Scheduler. That job runs the content through a two-stage pipeline: a passthrough extractor that strips HTML and hands the plain text to an LLM (Anthropic Claude by default), which returns a JSON array of named entities with confidence scores.

Those suggestions appear in a Gutenberg sidebar panel. The editor sees each entity — name, type (person, org, place, other), confidence percentage — and can approve or dismiss it. Approving a suggestion creates a `navne_entity` taxonomy term and assigns it to the post. From that point on, the first mention of that name in the post body gets linked to the term's archive page at render time, without touching the stored post content.

The render-time linking uses `the_content` filter and an object-cached link map per post. The cache is invalidated when a suggestion is approved or dismissed, or when the term is edited or deleted.

## What's in this release

- **Async LLM pipeline** — Action Scheduler job dispatches on publish, processes in the background, sets job status through queued → processing → complete/failed
- **LLM provider abstraction** — `ProviderInterface` with Anthropic implementation; adding a new provider means implementing one interface
- **Suggestions table** — Custom DB table (`wp_navne_suggestions`) with CRUD and status lifecycle (pending / approved / dismissed)
- **REST API** — Four endpoints under `navne/v1`: GET suggestions, POST approve, POST dismiss, POST retry
- **Gutenberg sidebar** — Four states (idle, processing, complete, failed), per-entity approve/dismiss cards, approved entities list, retry on failure, polls after save
- **Render-time linking** — First-mention linking, case-insensitive, double-link guard, object cache with targeted invalidation
- **`navne_entity` taxonomy** — Flat, public, non-hierarchical

---

## What's not in this release

This is a PoC. Several features are designed and documented but deliberately deferred:

- **Org definition list** — The LLM prompt has a placeholder for organization-specific definitions (local figures, acronyms, jargon). This is the first thing to build in v1.1.
- **Plugin settings page** — API key and provider are set via WP-CLI for now
- **Public taxonomy archive pages** — Terms are created and assigned but archive pages are unstyled
- **spaCy NER pre-filter** — The extractor passes full post content to the LLM; a candidate pre-filter is the right next NLP investment after the org definition list
- **Safe and YOLO modes** — v1.0 is Suggest mode only; Safe (whitelist-only auto-linking) and YOLO (fully automatic with mod queue) are future operating modes
- **Bulk re-indexing** — No tooling to process the existing article archive
- **Removing approved links** — Once approved, a link can only be removed by deleting the taxonomy term

## Known gaps

- Re-saving an article creates a new round of pending suggestions alongside any previously dismissed ones. The dismissed ones are preserved; the new pending ones layer on top. Fine for now, worth a proper dedup pass later.
- `wp_cache_flush_group` on term edit/delete is only effective with object cache backends that support group flushing (Redis, Memcached). On the default file-based cache, each key has to expire on its own.
